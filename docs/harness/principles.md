# Harness 12 Principles

> 코드베이스가 AI 에이전트에게 얼마나 친화적인지를 평가하는 12가지 원칙.
> OpenAI "Harness Engineering" 개념 기반. 각 원칙을 0-10점으로 채점한다.

## 목차

| # | 원칙 | 핵심 질문 |
|---|------|-----------|
| 1 | Agent Entry Point | AGENTS.md가 명확한 진입점인가? |
| 2 | Map, Not Manual | 문서가 지도(map)인가, 매뉴얼인가? |
| 3 | Invariant Enforcement | 실수를 도구가 자동으로 잡는가? |
| 4 | Convention Over Configuration | 명시적 규칙이 있는가? |
| 5 | Progressive Disclosure | 정보가 계층적으로 구성되어 있는가? |
| 6 | Layered Architecture | 의존성 방향이 명확한가? |
| 7 | Garbage Collection | stale 코드/문서를 주기적으로 정리하는가? |
| 8 | Observability | 에이전트가 작업 결과를 검증할 수 있는가? |
| 9 | Knowledge in Repo | 지식이 레포에 있는가? |
| 10 | Reproducibility | 동일 입력이 동일 결과를 보장하는가? |
| 11 | Modularity | 변경 영향 범위가 예측 가능한가? |
| 12 | Self-Documentation | 코드 자체가 의도를 설명하는가? |

---

## P1. Agent Entry Point

**정의**: 에이전트가 처음 진입했을 때 어디서 시작할지 안내하는 명확한 진입점 존재 여부.

**체크리스트**:
- [ ] 루트에 `AGENTS.md` / `CLAUDE.md` 또는 동등한 진입점 파일 존재
- [ ] 프로젝트 목적, 기술 스택, 디렉토리 구조 개요 포함
- [ ] 빌드/테스트/린트 명령어가 복사-붙여넣기 가능한 형태
- [ ] 에이전트 필수 규칙(금지/필수 패턴) 명시
- [ ] 하위 문서로의 링크가 제공되어 필요 시 심화 탐색 가능

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 진입점 파일 전무. README도 에이전트 관점 정보 없음. |
| 3 | README에 빌드/테스트 명령어는 있으나 에이전트 전용 안내 없음. |
| 5 | AGENTS.md 존재하지만 불완전하거나 최신 아님. |
| 8 | 진입점이 구조/빌드/테스트/규칙을 모두 포함. 하위 세부 안내 부족. |
| 10 | 루트 진입점 + 하위 디렉토리별 안내 완비. 정기 갱신됨. |

**안티패턴**: 거대한 단일 README, 구전 지식 의존, 6개월 전 stale 진입점, 외부 링크만 나열.

## P2. Map, Not Manual

**정의**: 문서가 순차 읽기 매뉴얼이 아니라 필요한 정보를 즉시 찾아가는 지도 역할을 하는지.

**체크리스트**:
- [ ] 주요 문서에 목차 + 앵커 링크
- [ ] 관련 문서 간 크로스 레퍼런스 존재
- [ ] 각 문서가 단일 주제에 집중
- [ ] 디렉토리 구조가 네비게이션 역할
- [ ] 일관된 용어 사용으로 검색 친화적

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 문서 거의 없거나 순차적 매뉴얼. 목차/크로스 레퍼런스 없음. |
| 3 | 긴 단일 파일에 모든 내용. 탐색 어려움. |
| 5 | 주제별 분리되어 있으나 상호 참조 부족. |
| 8 | 목차, 크로스 레퍼런스, 주제 분리 양호. 일부 탐색 경로 누락. |
| 10 | 완전한 지도. 어떤 질문이든 2단계 이내 탐색으로 답 도달. |

**안티패턴**: 소설형 문서, 2000줄 단일 파일, 고아 문서, 불일치 용어, 순환 참조.

## P3. Invariant Enforcement

**정의**: 핵심 규칙이 문서에만 있지 않고 도구(linter, CI, hook 등)로 자동 강제되는지.

**체크리스트**:
- [ ] 코드 스타일이 linter/formatter로 자동 강제
- [ ] CI에서 lint, type check, test 필수 실행
- [ ] pre-commit hook 설정
- [ ] 아키텍처 규칙이 도구로 강제 (import 방향 등)
- [ ] 규칙 위반 시 구체적 수정 안내 에러 메시지

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | linter/formatter/CI 전무. 구두 전달. |
| 3 | linter 있으나 CI 미강제. 로컬 선택 실행. |
| 5 | CI에서 lint+test 실행. 아키텍처 규칙은 수동 리뷰. |
| 8 | lint/format/type/test CI 필수. 주요 아키텍처 도구 강제. |
| 10 | 모든 invariant 도구 자동 강제 + pre-commit + 수정 가이드. |

**안티패턴**: 주석 규칙, 선택적 CI, 비밀 규칙, 느린 피드백, 불명확 에러.

---

## P4. Convention Over Configuration

**정의**: 프로젝트 규칙이 암묵적 합의가 아닌 명시적으로 문서화/코드 반영되어 있는지.

**체크리스트**:
- [ ] 파일/디렉토리 명명 규칙 문서화 또는 구조에서 추론 가능
- [ ] 새 기능 추가 시 따라야 할 템플릿/예제 존재
- [ ] API, 스키마, 에러 코드의 명명 규칙 일관적
- [ ] 설정 파일 최소화, 대부분이 convention으로 결정
- [ ] convention 위반을 자동 감지 가능

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 규칙 없음. 같은 프로젝트 내 케이스 혼재. |
| 3 | 일부 일관성 있으나 통일 convention 부재. |
| 5 | convention 존재하고 대부분 따르나 명시적 문서 없음. |
| 8 | convention 문서화 + 코드베이스 전반 일관. |
| 10 | 문서화 + 도구 강제 + 코드 생성기/템플릿. |

**안티패턴**: 혼합 스타일, 마법 문자열, 과도한 설정, 예외의 일반화.

---

## P5. Progressive Disclosure

**정의**: 정보가 한꺼번에 쏟아지지 않고 작업 깊이에 맞춰 계층적으로 구성되어 있는지.

**체크리스트**:
- [ ] 정보가 최소 3단계 (개요 → 모듈 상세 → 엣지 케이스/ADR) 계층
- [ ] 각 디렉토리에 해당 범위 문서 존재
- [ ] 코드 주석이 "왜(why)"에 집중
- [ ] ADR 또는 의사결정 기록 존재
- [ ] 현재 작업 범위에서 불필요한 정보에 노출되지 않음

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 계층 없음. 루트 README 하나 또는 코드 주석에만 존재. |
| 3 | 루트 + 일부 모듈 문서. 깊이 간 연결 부족. |
| 5 | 2단계 구조. ADR/의사결정 기록 없음. |
| 8 | 3단계 이상 + ADR. 대부분 작업에서 적시 정보 발견. |
| 10 | 완전한 progressive disclosure. 어떤 깊이든 과부족 없음. |

**안티패턴**: 500줄 README, 문서 없는 모듈, 평면적 docs/, 의사결정 블랙홀.

---

## P6. Layered Architecture

**정의**: 의존성 방향이 명확해서 한 레이어 수정 시 영향 범위를 예측할 수 있는지.

**체크리스트**:
- [ ] 레이어 간 의존성 방향 문서화
- [ ] 순환 의존성 없음 또는 도구 감지
- [ ] 각 레이어의 책임/경계가 디렉토리에 반영
- [ ] 레이어 간 명확한 인터페이스
- [ ] 의존성 위반 자동 감지 도구 설정

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 레이어 개념 없음. 단일 디렉토리. 무질서. |
| 3 | 디렉토리 분리 있으나 경계 불분명. 순환 의존성. |
| 5 | 레이어 구분, 대부분 올바르나 일부 위반. 도구 강제 없음. |
| 8 | 명확한 레이어 + 문서화 + 대부분 도구 강제. |
| 10 | 완전한 레이어. 의존성 도구 강제. 순환 zero. |

**안티패턴**: God 모듈, 레이어 관통, 양방향 의존, 비대한 utils/.

---

## P7. Garbage Collection

**정의**: 미사용 코드, 오래된 문서, deprecated API를 주기적으로 정리하는지.

**체크리스트**:
- [ ] dead code 탐지 도구 CI 통합
- [ ] deprecated 코드에 `@deprecated` + 대안 명시
- [ ] feature flag 주기적 정리
- [ ] TODO/FIXME 관리 (이슈 번호 연결)
- [ ] 문서가 코드 변경에 맞춰 갱신

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | dead code, stale 문서 대량. 유효 코드 판단 불가. |
| 3 | deprecated 일부 표시. 실제 정리 미수행. |
| 5 | 가끔 정리. @deprecated 사용. 자동 탐지 없음. |
| 8 | dead code 도구 + 주기적 정리 + TODO에 이슈 링크. |
| 10 | 자동 GC. dead code CI 경고. feature flag 만료일. |

**안티패턴**: 주석 처리된 코드 묘지, 2년 전 TODO, 좀비 feature flag, 시간 캡슐 문서.

---

## P8. Observability

**정의**: 에이전트가 작업 결과를 스스로 검증할 수 있는 피드백 메커니즘 존재 여부.

**체크리스트**:
- [ ] 모듈 단위로 빠른 테스트 실행 가능
- [ ] 테스트 실행 명령어 간단하고 문서화
- [ ] type check 명령어 존재 + 빠른 실행
- [ ] 빌드 실패 시 구체적 원인 특정 가능
- [ ] 로컬에서 변경 결과 즉시 확인

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 테스트 없음. 검증 수단 전무. |
| 3 | 테스트 있으나 복잡하거나 느림 (10분+). 부분 실행 불가. |
| 5 | 테스트 + type check. 모듈 단위 빠른 검증 어려움. |
| 8 | 모듈 단위 테스트 + type check + lint 빠른 실행. |
| 10 | 수초 내 피드백. 파일 단위 테스트, 증분 빌드, 명확 에러. |

**안티패턴**: 전체 스위트만 실행 가능, assertion 없는 빈 테스트, 환경 의존, flaky test.

---

## P9. Knowledge in Repo

**정의**: 프로젝트 지식이 팀원 머릿속이나 외부(Slack, Notion)가 아닌 레포 자체에 존재하는지.

**체크리스트**:
- [ ] 아키텍처 결정 이유가 ADR 또는 주석으로 기록
- [ ] 비즈니스 규칙 출처/근거 문서화
- [ ] 환경 설정, 배포 절차, 트러블슈팅 가이드 레포 내 존재
- [ ] 외부 서비스 연동 제약/특이사항 코드 근처 문서화
- [ ] 온보딩에 필요한 모든 정보가 레포만으로 충족

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 대부분 특정 팀원 머릿속에만 존재. |
| 3 | 기본 설정 문서 있으나 아키텍처 결정 근거 외부에만. |
| 5 | 주요 문서 레포에 있으나 일부 중요 맥락 외부. ADR 없음. |
| 8 | ADR + 비즈니스 규칙 문서화 + 트러블슈팅. 일부 암묵 지식 누락. |
| 10 | 레포가 single source of truth. 외부 참조 불필요. |

**안티패턴**: 구전 전통, Slack 아카이브, Notion 의존, 부족 지식.

---

## P10. Reproducibility

**정의**: 동일 입력이 항상 동일 결과(빌드, 테스트, 배포)를 내는지.

**체크리스트**:
- [ ] 의존성 lock 파일로 버전 고정
- [ ] 결정론적 빌드 (동일 커밋 = 동일 결과물)
- [ ] 개발 환경 설정 자동화 (Docker, devcontainer 등)
- [ ] 테스트가 외부 상태(네트워크, DB, 시간)에 미의존
- [ ] CI와 로컬 환경 차이 최소화

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | lock 파일 없음. "내 컴에서는 되는데?" |
| 3 | lock 파일 있으나 환경 설정 수동. CI/로컬 차이 큼. |
| 5 | 의존성 고정 + 기본 환경 자동화. 일부 외부 의존 테스트. |
| 8 | 결정론적 빌드 + 컨테이너 + 대부분 격리 테스트. |
| 10 | 완전 재현성. 어떤 머신에서든 동일 결과. 테스트 완전 격리. |

**안티패턴**: `^18.0.0` 부유 의존성, snowflake 서버, 시간 의존 테스트, 암묵적 전역 상태.

## P11. Modularity

**정의**: 독립적으로 이해/수정 가능한 모듈 구성으로 변경 영향이 예측 가능한지.

**체크리스트**:
- [ ] 모듈이 명확한 public API + 캡슐화된 내부 구현
- [ ] 모듈 간 의존성 최소화 (낮은 결합도)
- [ ] 모듈 내부 높은 응집도
- [ ] 독립적 테스트 가능
- [ ] 새 기능 = 새 모듈 추가가 자연스러움

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | 모듈 개념 없음. 수천 줄 단일 파일. |
| 3 | 파일 분리는 있으나 경계 불명확. 1파일 수정 → 10파일 영향. |
| 5 | 모듈 구조 존재. 대부분 독립적. 일부 강결합. |
| 8 | 명확한 경계 + public API + 독립 테스트. |
| 10 | 완전 모듈화. 각 모듈 독립 이해/수정/테스트. 영향 범위 항상 예측. |

**안티패턴**: God 클래스, 전역 상태 남용, shotgun surgery, 깊은 상속 체인.

---

## P12. Self-Documentation

**정의**: 코드 자체가 별도 문서 없이 의도와 동작을 설명하는지.

**체크리스트**:
- [ ] 함수/변수/클래스 이름이 역할과 의도 전달
- [ ] 타입 시스템이 값의 범위를 코드로 표현
- [ ] 매직 넘버/스트링 → named constant
- [ ] 복잡한 비즈니스 로직에 "왜(why)" 주석
- [ ] 함수 크기 적절 (20-30줄), 함수명으로 동작 예측

**채점**:

| 점수 | 기준 |
|------|------|
| 0 | `a`, `tmp`, `data` 수준 이름. 타입 없음. 매직 넘버. |
| 3 | 이름 어느 정도 서술적이나 불일치 많음. 타입 부분적. |
| 5 | 대부분 서술적 이름 + 타입. 복잡 로직에 "왜" 주석 부족. |
| 8 | 서술적 이름 + 강한 타입 + named constant + "왜" 주석. |
| 10 | 코드 자체가 완전한 문서. 이름/타입/구조/주석 완벽 조화. |

**안티패턴**: 약어 남용, 거짓말하는 이름, 주석이 코드 반복, `any` 지옥, 500줄 함수.

## 정량 판정 보조 지표

주관 최소화를 위한 정량 기준. 체크리스트 정성 평가와 병행한다.

| 원칙 | 정량 지표 | 0점 | 5점 | 10점 |
|------|-----------|-----|-----|------|
| P1 | 체크리스트 5항목 충족 수 | 0개 | 2-3개 | 5개 |
| P2 | 문서 간 상호 참조 비율 | 0% | 30-60% | >90% |
| P3 | CI 자동 검증 항목 수 (lint/type/test/arch) | 0개 | 2개 | 4개 + pre-commit |
| P4 | 파일명 규칙 준수율 | <50% | 70-85% | >95% |
| P5 | 정보 계층 단계 수 | 0-1 | 2단계 | 3단계 + 각 완비 |
| P6 | 순환 의존성 수 | 측정 불가 | 2-4개 | 0개 + 도구 강제 |
| P7 | dead code / stale 문서 비율 | >20% | 5-10% | <2% + 자동 탐지 |
| P8 | 피드백 루프 소요 시간 | 없음 | 3-10분 | <1분 |
| P9 | 외부 전용 지식 비율 | >80% | 20-50% | <5% |
| P10 | 환경 재현 실패 빈도 | 매주 | 분기 1회 | 0회 |
| P11 | 1개 변경 시 영향 파일 수 (평균) | >10개 | 3-5개 | 1개 |
| P12 | 외부 문서 참조 필요 함수 비율 | >50% | 15-30% | <5% |

**사용법**: 체크리스트 정성 평가와 정량 지표를 교차 확인. 두 기준의 점수 차가 2점 이상이면 근거 재검토.
